Kod:
  def get_rates(currency: str, days: int) -> pd.DataFrame:
      end = date.today()
      start = end - timedelta(days=days)

      url = f"https://api.nbp.pl/api/exchangerates/rates/A/{currency}/{start}/{end}/?format=json"
      resp = requests.get(url, timeout=30)
      resp.raise_for_status()

      for r in resp.json().get("rates", []):
          rows.append({
              "Date": r["effectiveDate"],
              "Currency": currency,
              "Rate": r["mid"]
          })

      return pd.DataFrame(rows)

FUNKCJE DLT UŻYTE W LAB5
------------------------

1. sql_database() - Źródło danych
   from dlt.sources.sql_database import sql_database

   src = sql_database(connection_string, schema="Production")
   src.with_resources("Product", "ProductCategory")

2. pipeline() - Konfiguracja pipeline'u
   pipeline = dlt.pipeline(
       pipeline_name="aw_to_extract",
       destination="mssql",
       dataset_name="Extract"
   )

3. pipeline.run() - Wykonanie ładowania
   pipeline.run(source, write_disposition="replace")
   # write_disposition: "replace" = DROP + CREATE (zamiast APPEND)

4. write_disposition - Strategia ładowania
   - "replace":  DROP tabelę, utwórz nową (overwrite)
   - "append":   Dodaj wiersze (INSERT)
   - "merge":    Merge po kluczu głównym


METADATA TABELE TWORZONE PRZEZ DLT
-----------------------------------
DLT automatycznie tworzy pomocnicze tabele:

1. _dlt_loads
   - Loguje każde ładowanie
   - Zawiera: load_id, schema_name, status, insert_date
   - Używane do tracking'u i audit'u

2. _dlt_pipeline_state
   - Stan pipeline'u (wersje, konfiguracja)
   - Helps przy inkrementalnym ładowaniu

3. _dlt_version
   - Wersja dlt engine'u
   - Historija zmian schematu

PATCH APLIKOWANY W LAB5
-----------------------
Problem: Domyślnie dlt próbuje połączyć się przez TCP port 1433
Rozwiązanie: Patch modyfikujący ODBC DSN aby pominąć port
            (pozwala na połączenie przez shared memory / named pipes)

Kod patchu (sql_extract.py):
  from dlt.destinations.impl.mssql import configuration as _mssql_cfg

  _orig_odbc_dict = _mssql_cfg.MsSqlCredentials.get_odbc_dsn_dict

  def _odbc_dict_no_port(self):
      d = _orig_odbc_dict(self)
      d["SERVER"] = self.host  # Remove port
      return d

  _mssql_cfg.MsSqlCredentials.get_odbc_dsn_dict = _odbc_dict_no_port

5. OPIS DDT (DATA DEFINITION TRANSFER)
================================================================================

Co to jest DDT?
---------------
DDT (Data Definition Transfer) to technika przesyłania metadanych:
  - Struktury tabel (CREATE TABLE)
  - Definicje kolumn (typy, constrainty)
  - Indeksy, klucze, relacje
  - Trigger'y, procedures
  - Viewsy

W kontekście LAB5:
  DDT w dlt oznacza automatyczne przesyłanie definicji tabel ze źródła do celu.


JAK DZIAŁA DDT W DLT?
---------------------

Krok 1: Schema Reflection
  - DLT odczytuje INFORMATION_SCHEMA tabeli źródłowej
  - SELECT ... FROM sys.tables, sys.columns, sys.types
  - Zawiera: nazwy kolumn, typy, nullable, primary keys

Krok 2: Mapowanie typów
  - SQL Server INT → dlt bigint
  - SQL Server VARCHAR → dlt text
  - SQL Server DATETIME → dlt timestamp

Krok 3: Tworzenie tabeli docelowej
  - DLT generuje CREATE TABLE statement
  - Tworzy tabelę w [Extract] schemacie

Krok 4: Dodanie kolumn systemowych dlt
  - _dlt_load_id   (śledzenie ładowania)
  - _dlt_id        (unique identifier)
  - _dlt_timestamp (kiedy załadowano)

Przykład wygenerowanej tabeli:
  CREATE TABLE [extract].[product] (
    _dlt_load_id                    TEXT,
    _dlt_id                         TEXT,
    _dlt_timestamp                  TIMESTAMP,
    product_id                      BIGINT,
    name                            TEXT,
    product_number                  TEXT,
    make_flag                       BIT,
    finished_goods_flag             BIT,
    color                           TEXT,
    safety_stock_level              BIGINT,
    reorder_point                   BIGINT,
    standard_cost                   NUMERIC,
    list_price                      NUMERIC,
    size                            TEXT,
    ...
  );


ZALETY DDT
----------
✓ Automatyczne synchronizowanie schematów
✓ Nie trzeba ręcznie pisać CREATE TABLE
✓ Zmapowanie typów danych odbywają się automatycznie
✓ Śledzenie zmian schematu (_dlt_version)
✓ Obsługa złożonych typów (arrays, JSON)


WYZWANIA DDT
-----------
✗ Nieznane typy SQL (MONEY, GEOGRAPHY) generują WARNING'i
✗ Pewne constrainty mogą być pominięte
✗ Relacje (FOREIGN KEY) się przesyłają ale mogą być ignorowane
✗ Indeksy nie są przesyłane automatycznie


WARNINGS ZAOBSERWOWANE W LAB5
------------------------------
[WARNING] Unknown data type MONEY
  - Rozwiązanie: automatycznie mapowane na decimal

[WARNING] Unknown data type GEOGRAPHY
  - Rozwiązanie: mapowane na text/JSON

[WARNING] Unknown data type HIERARCHYID
  - Rozwiązanie: mapowane na text
